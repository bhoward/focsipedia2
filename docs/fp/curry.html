Currying

When we write a function that takes multiple arguments, we may list the parameters in parentheses, separated by commas:

val average = (a: Int, b: Int) => (a + b) / 2.0
// average: Function2[Int, Int, Double] = repl.MdocSession$MdocApp$$Lambda/0x00000001284f9000@2216d7c5
average(7, 10)
// res0: Double = 8.5

This is actually a luxury! We could make do entirely with functions taking only a single argument.

Of course, one way to do this is to make use of tuples, but a more interesting solution is to take advantage of higher-order functions. Based on the example above, we could define average2 as a function that first takes a value for the parameter a and then returns another function that expects a value for b:

val average2 = (a: Int) => (b: Int) => (a + b) / 2.0
// average2: Function1[Int, Function1[Int, Double]] = repl.MdocSession$MdocApp1$$Lambda/0x00000001284f8c00@f4d2b
average2(7)(10)
// res2: Double = 8.5

Now when we apply average2 to 7, the result is a function value that is then applied to the 10.

Here is the same code, written out more explicitly:

val average2: Int => (Int => Double) = (a: Int) => {
  (b: Int) => {
    (a + b) / 2.0
  }
}
// average2: Function1[Int, Function1[Int, Double]] = repl.MdocSession$MdocApp3$$Lambda/0x00000001284faab0@610b269a
val result1: Int => Double = average2(7)
// result1: Function1[Int, Double] = repl.MdocSession$MdocApp3$$Lambda/0x00000001284fb0b0@42f3e42e
val result: Double = result1(10)
// result: Double = 8.5

This replacement of multiple-parameter functions with a sequence of single-parameter functions is called currying, named after the logician Haskell B. Curry.1 One advantage of this, other than the simplicity of only needing one kind of function, is that it is often useful to create a partially applied function, where some of its arguments have been supplied to create a new function ready to be given the rest. For example, suppose we have a function for formatting exam grades:

val format_grade = (exam: String, total: Int, name: String, points: Int) => {
  name + ", " + exam + ": " + points + "/" + total
}
// format_grade: Function4[String, Int, String, Int, String] = repl.MdocSession$MdocApp3$$Lambda/0x00000001284fb8f8@6bb972a0
format_grade("Midterm", 100, "Brian", 93)
// res4: String = "Brian, Midterm: 93/100"

Scala has a built=in operation on functions, .curried, that produces the curried version of a function:

val curried_format_grade = format_grade.curried
// curried_format_grade: Function1[String, Function1[Int, Function1[String, Function1[Int, String]]]] = scala.Function4$$Lambda/0x00000001284de7a8@4616cc0d

We could take advantage of currying to create a specialized function for formatting the midterm grades:

val format_midterm = format_grade.curried("Midterm")(100)
// format_midterm: Function1[String, Function1[Int, String]] = scala.Function4$$Lambda/0x00000001284def58@5d99152e
format_midterm("Brian")(93)
// res5: String = "Brian, Midterm: 93/100"
format_midterm("Alice")(97)
// res6: String = "Alice, Midterm: 97/100"

The first two arguments of format_grade have been provided with the exam name ("Midterm") and the total number of points (100). Now we have a new function, bound to format_midterm, that just needs to be applied to a student name and grade, and then it can produce a string with all four components.

Tuples and Parameters​

It might seem that tuples should be used to pass multiple parameters to functions, but for reasons of compatibility with Java, Scala gives special treatment to multiple-parameter functions. We can force it to pass tuples of arguments, and bind them to tuples of parameters, by including an extra pair of parentheses:

def f(p: (String, Int)): String = {
  val (name, points) = p
  name + ": " + points + "/100"
}
f(("Brian", 93))
// res8: String = "Brian: 93/100"

Now, that's ugly, and unless you really need to do that, don't do it. However, this brings up an interesting equivalence of types. Note that the type for f here is ((string, int)) => string; in terms of sets, this is the set of functions stringstring×int\text{string}^{\text{string}\times\text{int}}stringstring×int. Compare this with the equivalent ("curried") function g:

val g: String => Int => String = (name) => (points) => {
  name + ": " + points + "/100"
}
// g: Function1[String, Function1[Int, String]] = repl.MdocSession$MdocApp7$$Lambda/0x00000001284fcd10@6a7eef40
g("Brian")(93)
// res9: String = "Brian: 93/100"

The type of g here is string => int => string; in terms of sets, this is the set (stringint)string(\text{string}^{\text{int}})^{\text{string}}(stringint)string. If these types are truly equivalent, in the sense that every function in one corresponds to a unique function in the other, then that suggests that there might be a general equivalence of the form

AB×C≡(AC)BA^{B\times C}\equiv(A^C)^BAB×C≡(AC)B

This is indeed true (and it should remind you of a corresponding fact about exponents from ordinary algebra), and we can write the functions in Scala that mediate this equivalence:

def curry[A, B, C](f: ((B, C)) => A): B => C => A = {
  (b: B) => (c: C) => f((b, c))
}
def uncurry[A, B, C](g: B => C => A): ((B, C)) => A = {
  (p: (B, C)) => g(p._1)(p._2)
}

That is, given any function from the pair type (B, C) to A (A, B, and C are type parameters), we can apply the curry function to it to get the corresponding curried function of type B => C => A. The uncurry function is the inverse of this. Since we have functions going each direction that are inverses to each other, this shows that the two types (or sets) are equivalent.

  1. As is often the case when things are named, Curry did not originate this idea. He got it from Moses Schönfinkel, who may have picked it up from Gottlob Frege, but "currying" is easier to say than "schönfinkeling" or "fregeing"…. ↩
